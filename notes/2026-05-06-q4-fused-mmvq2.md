# Qwen3.6 Q4_0 Fused MMVQ2

Date: 2026-05-06

## Summary

Implemented a performance-positive llama.cpp/SYCL fused adjacent Q4_0 MMVQ pair path behind `GGML_SYCL_FUSE_MMVQ2=1`.

The fused path computes adjacent same-input Q4_0 x Q8_1 matvec pairs in one SYCL kernel launch, using llama.cpp's existing reordered Q4_0/Q8_1 DP4A vecdot code. It does not change quantization, KV dtype, sampling, speculative decode, or power limits.

Quality status: single-card PPL matched exactly with fused off/on. A forced non-conversation 3-card deterministic generation check also matched byte-for-byte. However, 3-card PPL is not a stable oracle yet because the fusion-disabled baseline itself varied materially across repeat runs. Treat `GGML_SYCL_FUSE_MMVQ2=1` as experimental until a better multi-GPU logit/token harness is in place.

## Interpretation

The fused pair path is a real but modest end-to-end improvement. It confirms that repeated adjacent matvec launch overhead is part of the decode cost, but not the only limiter.

Because the multi-GPU PPL harness is nondeterministic even with fusion disabled, performance results remain useful, but quality validation for the 3-card path needs a dedicated logit/token comparison harness before making this a default recommendation.

The 4x result improved slightly, yet remains materially slower than 3x. That keeps the current hypothesis intact: 4x loses to added per-device graph/dispatch/sync overhead and small-collective overhead, not only to the Q4_0 ffn gate/up pair launches.

## Next Work

- Add a correctness harness that compares logits or generated token IDs between gate-off and gate-on for a fixed prompt without relying on the current multi-GPU PPL path.
- Test a more aggressive ESIMD Q4_0 x Q8_1 fused2 kernel inside llama.cpp after the DP4A fused launch path is stable.
- Consider fusing the following `GLU` for `ffn_gate/up -> SWIGLU`, but only after confirming buffer ownership and graph scheduling safety.
- Recheck 4x with `p512/n512` after any further local-kernel reduction; do not resume root-order sweeps without a new implementation change.
